# TDDD (TODO Driven Development)

## What This Is

TDDD (TODO Driven Development) is a record of a **practical development operating method**
for a specific era’s generative‑AI environment.

> This document is not a tutorial for building web pages.
> TDDD is a development operating method for leaving decisions in TODOs.

It does not propose new ideology or design theory.

---

## Positioning

TDDD is not any of the following:

- Part of VCDesign  
- An extension of VC‑AD  
- A permanent development framework  
- An independent methodology (◯◯DD)

TDDD is

> **“An operational procedure for safely running a designed world.”**

---

## Why This Is Recorded

With the spread of generative AI, the assumptions of development changed.

- Implementation is automatically generated
- Code can be discarded easily
- Decisions and implementation became separated

In this situation, TDDD is recorded to preserve
**how development was run at that time**.

---

## Why It Is Not Independent

TDDD depends heavily on:

- Tool configuration
- Differences in AI capability
- IDE integration status

Therefore, it should not be treated as a long‑term, fixed ideology
or design principle.

It is appropriate to keep it as an **annual operating manual**.

## Relationship to VCDesign / VC‑AD

- VCDesign: how to design value and responsibility
- VC‑AD: a map of boundaries that are often missed in the AI era
- TDDD: how to run development assuming that design and map

TDDD **assumes higher‑level concepts but does not contain them**.

---

## About the Future

In the future, if:

- IDE AI further integrates design mode and implementation mode, and
- the way TODO creation and implementation are switched changes

then TDDD’s form may change, or it may become unnecessary.

At that point,
**this record itself will be evidence that its role has ended**.

---

## Summary

TDDD is not an ideology.  
It is not a framework.

It is **a record of how development was operated in a certain era**.

Nothing more, nothing less.

# TDDD 簡単編（2026）

## これは何か

このドキュメントは  
**TODO を使って開発を進めるための最低限の手順書** です。

設計理論や思想の説明はしません。  
「どうやれば、壊れにくいものを作れるか」だけを書きます。

---

## 前提

- 何を作るかは大体決まっている
- 設計や分割が苦手
- 生成AIを使っている（ChatGPT / Claude / Copilot 等）
- VSCodeを使っている（HTMLやPythonが書けなくても構わない）

それで十分です。

---

## 重要な考え方（これだけ覚える）

- **コードを書き始めない**
- **最初に TODO を作る**
- **問題が起きたらコードではなく TODO を直す**

以上です。

---

## 全体の流れ（4ステップ）

1. TODO を作る
2. TODO を実行させる
3. 動かして確認する
4. TODO を直す

これを繰り返します。

---

## Step 1：TODO を作る

### やること

判断フェーズのAI操作で、こう頼みます。

これから作りたいものの TODO を作ってください。
やらないことも含めて整理してください。


### コツ

- 仕様が曖昧でも OK
- 分からない点は「未確定」と書かせる
- **コードはまだ書かせない**

### 成果物

- `TODO.md`

---

## Step 2：TODO フォルダを作る

プロジェクトに以下のフォルダを作ります。

todo/
└─ TODO.md


必要になったら、ここに増やします。

最初は `TODO.md` だけでよい。

慣れてきたら、確認条件を分けて `CHECK.md` に書いてもよい。

```text
todo/
├─ TODO.md
└─ CHECK.md
```

CHECK.md には、実装後に人間が見るポイントを書く。

---

## Step 3：TODO を実行させる

実行フェーズのAI操作で、こう頼みます。

todo/TODO.md に基づいて HTML と CSS でWebページを作ってください。
TODO に書かれていないことは実装しないでください。


### 重要

- **TODO 以外を根拠に実装させない**
- 言語やツールは TODO に明示する
- 勝手に補完し始めたら止める

---

## Step 4：動かして確認する（人間の仕事）

自分で実行します。

見るポイントは 2 つだけです。

- 動くか
- 期待と違わないか

---

## Step 5：おかしかったら TODO を直す

### やってはいけないこと

- いきなりコードを直す
- AI に「いい感じに直して」と言う

### 正しいやり方

1. 何が違ったかを書く
2. それを TODO に追加する

例：

```md
- [ ] 入力が空のときにエラーになるが、エラーではなく空結果を返したい
```

## Step 6：もう一度実装させる

同じ指示を出します。

```md
更新された TODO.md に基づいて実装を修正してください。
```

これを繰り返す

TODO を作る

実装する

確認する

TODO を直す

これだけです。
